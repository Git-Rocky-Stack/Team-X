## 4. The validation arc — why this took 2 rounds

### 4.1 Round 1 — release.yml run [25767951619](https://github.com/Git-Rocky-Stack/Team-X/actions/runs/25767951619) on `v3.2.1-test1` = `3e520a8`

**Surface symptoms:**

| Job | Result | Duration |
|---|---|---|
| Build macos-latest | ✅ success | 2m32s |
| Build ubuntu-latest | ✅ success | 3m57s |
| Build windows-latest | ✅ success | 6m44s |
| Create GitHub Release | ❌ failure at "Generate consolidated SHA256SUMS.txt" | 18s |

**Root cause: `actions/upload-artifact@v4` strips the longest common path prefix from `path:` patterns.**

The failure log was:
```
Expected exactly one version dir under artifacts/release/, found:
ls: cannot access 'artifacts/release/': No such file or directory
##[error]Process completed with exit code 1.
```

The new consolidated SHA step assumed merged artifacts would land at `artifacts/release/<version>/Team-X-...`. The reality, confirmed by `gh run download 25767951619 --name release-linux`, was `<version>/Team-X-...` inside the bundle (no `release/` prefix), so after `download-artifact merge-multiple: true` extracted everything into `artifacts/`, the actual layout was `artifacts/<version>/Team-X-...`.

This is **undocumented (or near-undocumented) behavior in `upload-artifact@v4`**: when every entry in the `path:` list shares a common directory prefix (`release/**/*.exe`, `release/**/*.dmg`, etc.), the action computes the longest common prefix (`release/`) and strips it before storing files in the artifact bundle. The intent is to avoid arbitrarily deep nested directory chains in artifacts — but the side effect is that any script that consumes the artifact must NOT assume the path it uploaded is preserved verbatim.

The defensive check in the SHA step (`if [ ${#dirs[@]} -ne 1 ]`) at least caught the empty result cleanly and dumped a diagnostic `ls` before exiting, rather than producing a corrupted manifest. That defensive shape made the diagnosis a 30-second exercise.

**No draft release was created** — `Create Release` is the next step after `Generate consolidated SHA256SUMS.txt`, and the failure short-circuited before it ran. So the round 1 cleanup was a simple `git push --delete v3.2.1-test1` + local tag delete; no `gh release delete` call was needed.

## 5. Patterns / lessons / drift guards added this session

### 5.1 `actions/upload-artifact@v4` strips the longest common path prefix — new gotcha

This is the round 1 bug's root cause and deserves first-class memory. When every pattern in `upload-artifact@v4`'s `path:` list shares a common directory prefix, that prefix is silently stripped from the stored paths. Consumer scripts that assume "I uploaded `release/3.2.0/foo.exe`, so I'll find it at `artifacts/release/3.2.0/foo.exe` after download-artifact" are wrong — they'll find it at `artifacts/3.2.0/foo.exe`.

The behavior is hinted at in the upload-artifact docs ("the directory structure is preserved relative to the workspace") but the specific common-prefix-stripping is not called out explicitly. The only reliable way to know what you'll get is to push a real run and inspect with `gh run download <run-id> --name <bundle>`.

**Drift guard (now in memory):**
- Post-merge scripts: use depth-agnostic patterns (`artifacts/**/<file>` with `globstar` enabled) OR glob the next-level dir (`artifacts/*/`) OR explicitly `ls -la artifacts/` first to confirm layout.
- When adding/modifying upload-artifact `path:` lists, verify the actual stored layout on the next run (`gh run download` is cheap and authoritative). Never rely on the documented behavior alone.
- Memory entry `[[project-v3-2-1-release-workflow-followups]]` now carries this gotcha permanently.

### 5.2 Defensive checks pay for themselves in diagnosis speed

The consolidated SHA gen step in `3e520a8` had a `if [ ${#dirs[@]} -ne 1 ]` check that printed a diagnostic and `ls -la artifacts/release/` before exiting. When round 1 failed, the failure log contained the literal `artifacts/release/` path, the empty dir-list, AND the `ls: cannot access` confirmation — three independent signals pointing at the same root cause. Diagnosis took under a minute.

A naive `cd artifacts/release/*/` without the check would have failed with `cd: too many arguments` (bash error, no glob expansion) or `cd: No such file or directory`, neither of which clearly says "the dir doesn't exist at all". The defensive shape paid for its 5 extra lines of script.

**Pattern:** any path that's computed from external state (artifact downloads, glob expansion, environment vars) should have a sanity check that names the path it's looking for. Fail fast with a diagnostic; don't proceed with `cd` that might silently succeed in the wrong dir.

### 5.3 Test tags are the right validation primitive for `release.yml` changes

`release.yml` only fires on tag push. Changes to it cannot be exercised by normal `ci.yml` runs or even by merging the PR — they only run when a tag exists. The pattern this session used:

1. Make the change on `main` directly (or via PR + squash merge)
2. Push a `vX.Y.Z-testN` tag pointing at the change
3. Watch `release.yml`
4. Validate the draft
5. Clean up tag + draft (`gh release delete vX.Y.Z-testN --cleanup-tag --yes`)
6. When ready, push the real `vX.Y.Z` tag

The test-tag iteration was cheap — round 1 took ~7 minutes total elapsed, round 2 took ~7m30s. Compared to the cost of debugging a botched real-version release in production (which is what v3.2.0 was), this is a no-brainer.

**Heuristic:** any release.yml change should go through at least one test-tag validation before the real version tag, especially when the change touches the publish/upload path. The cost is ~10 min of CI; the avoided cost is "hand-fixing a botched release" (which v3.2.0 did to its tune of ~10 min of manual draft cleanup).

### 5.4 `gh release delete --cleanup-tag --yes` is the right cleanup primitive

The first instinct on cleanup might be `gh release delete <tag>` followed by `git push --delete origin <tag>` and `git tag -d <tag>`. The `--cleanup-tag` flag on `gh release delete` collapses the first two into one network call, which is both faster and atomic (no risk of release-deleted-but-tag-orphaned states). Local tag still has to be deleted separately with `git tag -d`.

**Memory:** when cleaning up a test-tag release, `gh release delete <tag> --cleanup-tag --yes` + `git tag -d <tag>` is the two-command cleanup. Don't separate the release and tag deletions.

### 5.5 Backfilling old releases with new asset types works retroactively

The §7.6 backfill — uploading `latest*.yml` + `.blockmap` to the existing published v3.2.0 release — worked exactly as intended. electron-updater's auto-update discovery just reads `<repo>/releases/latest`'s assets via GitHub's API; it doesn't care whether those assets were uploaded at release time or 5 hours after. So users on v3.2.0 will see v3.2.1 (when it ships) as an available update via "Check for Updates" in Settings, even though v3.2.0 was tagged before the `latest*.yml` files existed.

**Limitation:** the cached per-OS artifact bundles from the original `release.yml` run had a 7-day retention window. The v3.2.0 backfill was performed inside that window (2026-05-12, with bundles expiring 2026-05-19). For older releases past the retention window, backfilling would require re-running the build locally and uploading manually. Future release.yml runs producing the same manifests will be backfillable for 7 days from their respective runs.
