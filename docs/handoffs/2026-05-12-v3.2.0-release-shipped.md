## 4. The release.yml debug arc — why this took 3 rounds + a manual finish

Each round resolved one or two failure modes; each fix exposed the next layer. The total elapsed time across the three release.yml runs was ~22 + ~30 + ~25 = **~77 minutes of build time**, plus ~30 minutes of diagnosis between rounds. Manual draft cleanup added another ~10 minutes after the third (green) run.

### 4.1 Round 1 — release.yml run [25705728157](https://github.com/Git-Rocky-Stack/Team-X/actions/runs/25705728157) on `818b1cb`

**Surface symptoms:**

| Platform | Reported step | Reality |
|---|---|---|
| ubuntu-latest | ❌ Lint + Typecheck + Test | 30+ TS6305 errors at typecheck. Step correctly reports failure. |
| macos-latest | ❌ Lint + Typecheck + Test | Same TS6305 cascade. Step correctly reports failure. |
| windows-latest | ✅ Lint + Typecheck + Test (!) | **Step lies.** Typecheck failed with exit 2 (same TS6305 cascade), but the combined run-block continued to `pnpm test` which succeeded, and pwsh returned only the last exit code. The step shows green. The actual failure was at `Build + Package (win)` with `cannot find specified resource "build/installerBanner.bmp"`. |

**Two stacked root causes:**

1. **Workspace pre-build missing.** `apps/desktop/tsconfig.*.json` declares each of the 5 sibling packages (`shared-types`, `role-schema`, `provider-router`, `telemetry-core`, `intelligence`) as a composite project reference. `tsc --noEmit` reads the referenced project's `dist/index.d.ts` from disk. CI starts from a clean checkout with no `dist/` (gitignored), so without an explicit `tsc --build` pass for those 5 packages, the typecheck step sees stale (i.e. nonexistent) composite outputs and emits a TS6305 cascade. `ci.yml` had fixed this in `c09cc15` ("ci: pre-build workspace packages before typecheck"). `release.yml` never got the same step. The cross-platform matrix in `ci.yml` was masking the gap — the green daily CI hid the fact that the release workflow had latent bit rot.

2. **`installerBanner.bmp` whitelisted in `.gitignore` but never tracked.** `.gitignore` line 14 globs out `apps/desktop/build/*`; lines 15–21 whitelist 7 specific assets. `installerBanner.bmp` was not on the whitelist. The 154 KB BMP existed in Rocky's working tree (May 1 mtime) but never made it into git. Local `pnpm dist:win` worked; CI did not. The x64 NSIS pass apparently succeeded with the same config because electron-builder's resource resolution short-circuits per arch — the missing-banner check fires the second time around (arm64).

**The Windows "step lies" issue is structurally important.** Combining `Lint + Typecheck + Test` into a single multi-line `run:` block worked under `bash -e` on Linux/macOS (first failure aborts the step) but not under pwsh on Windows (each command runs in sequence; only the last exit code reaches the runner). A green check on Windows hid a real typecheck failure for the entire pre-launch-readiness era. The fix — splitting into three explicit steps — makes the failure mode visible per command and removes pwsh's right to lie about a multi-step shell.

### 4.2 Round 2 — release.yml run [25706509574](https://github.com/Git-Rocky-Stack/Team-X/actions/runs/25706509574) on `ee4b54a`

**Surface symptoms:**

| Platform | Reported step | Reality |
|---|---|---|
| ubuntu-latest | ❌ Generate SHA256 checksums | `syntax error near unexpected token '2'`. Build + Package succeeded; SHA generation failed. |
| windows-latest | ❌ Generate SHA256 checksums | Same bash syntax error. Build + Package succeeded — including both x64 and arm64 NSIS installers. |
| macos-latest | ❌ Build + Package (mac) | `⨯ /Users/runner/work/Team-X/Team-X/apps/desktop not a file`. Build + Package never produced a .dmg. |

**Two unrelated root causes that surfaced in the same run:**

1. **`for f in *.exe *.dmg ... 2>/dev/null; do` is a bash syntax error.** The intent was to suppress glob-no-match errors. The implementation was nonsense — globs don't write to stderr, and bash doesn't accept redirections on the `for` keyword in that position. Fix: `shopt -s nullglob` makes unmatched patterns expand to empty (the documented bash mechanism for "glob no-match → don't iterate"). Pattern list extended to include `mac/*.dmg` and `mac-arm64/*.dmg` because electron-builder writes macOS images into per-arch subdirectories under `release/${version}/`, not into the version directory directly. Hardened with `set -euo pipefail` so future bugs surface.

2. **electron-builder + empty `CSC_LINK` env var = boom.** The Phase 4 scaffolding from `07cdea2` set five signing env vars from repo secrets. Missing secrets in GitHub Actions render as **empty strings**, not unset. electron-builder's signing path treats `process.env.CSC_LINK` truthiness inconsistently — empty string in some code paths means "use default", and the default falls back to the current working directory. When cwd is `apps/desktop` (a directory, not a file), the cert loader fails with `not a file`. The predecessor handoff predicted "graceful degradation to unsigned .dmg", but that was theoretical — the cross-platform matrix was red until 2026-05-11 so the empty-secrets path had never been exercised. Fix: `mac.identity: null` in `electron-builder.yml` explicitly disables signing per the documented contract. When Phases 1–3 of the Mac signing plan complete and real secrets land in repo Settings, removing `identity: null` re-enables signing automatically via the env-var path.

### 4.3 Round 3 — release.yml run [25707677222](https://github.com/Git-Rocky-Stack/Team-X/actions/runs/25707677222) on `b641e7d`

**Surface symptoms:**

| Job | Result |
|---|---|
| Build ubuntu-latest | ✅ success |
| Build macos-latest | ✅ success |
| Build windows-latest | ✅ success |
| Create GitHub Release | ✅ success |

**All four jobs green. Draft release created with 8 assets.** First success on the cross-platform release pipeline ever.

But the draft contained two defects discovered during verification:

1. **`SHA256SUMS.txt` was incomplete.** It contained only 4 hashes — `Team-X-3.2.0-x86_64.AppImage`, `Team-X-3.2.0-amd64.deb`, `builder-debug.yml`, `latest-linux.yml`. Missing: all 5 Windows + macOS installer hashes. Root cause: each of the 3 OS jobs generates its own `SHA256SUMS.txt` covering only that OS's artifacts. All three upload via `actions/upload-artifact@v4` into bundles named `release-${platform}`. The `publish` job then runs `actions/download-artifact@v4` with `merge-multiple: true` — **which flattens all 3 bundles into one `artifacts/` directory, overwriting same-named files**. Only the last-extracted `SHA256SUMS.txt` survives. In this run, Linux's file won the merge race.

2. **Stray `Team-X.exe` asset (181 MB).** The publish step's upload glob was `artifacts/**/*.exe`. electron-builder's Windows build produces both the installer (`Team-X-3.2.0-Setup-x64.exe`, `Team-X-3.2.0-Setup-arm64.exe` — what we ship) and the unpacked binary (`win-unpacked/Team-X.exe`, `win-arm64-unpacked/Team-X.exe` — internal). The glob matched both. The unpacked binary landed in the release as a confusingly-named asset just called `Team-X.exe`.

## 5. Patterns / lessons / drift guards added this session

### 5.1 `release.yml` and `ci.yml` MUST mirror each other for shared steps

The TS6305 cascade in round 1 was the symptom of a 2-week structural drift: `ci.yml` got the `Build workspace packages (composite refs)` step on 2026-04-28 (`c09cc15`); `release.yml` never did. The cross-platform matrix in `ci.yml` was the only thing exercising the typecheck/test paths, so the daily green CI gave a false sense of "release-ready" while `release.yml` had been latently broken.

**Drift guard:** any change to the Lint/Typecheck/Test pipeline in `ci.yml` must be mirrored to `release.yml` in the same commit. The shared steps now are: `Install Electron runtime libraries (Linux, with libsecret-1-0)`, `Install dependencies`, `Build workspace packages (composite refs)`, `Lint`, `Typecheck`, `Test`. Comments in both workflows now reference each other ("Mirrors the ci.yml check job — keep both in sync.").

### 5.2 `bash -e` ≠ `pwsh` for multi-line shell scripts

GitHub Actions `run:` blocks on Windows default to pwsh. pwsh does **not** abort a multi-line script on first command failure — it runs every command and surfaces only the last exit code as the step's exit code. The classic `pnpm lint && pnpm typecheck && pnpm test` chain (which works fine inline on bash with `-e`) silently lies on Windows when written as separate lines.

**Heuristic:** if you have a multi-step shell sequence in a workflow and the cross-platform matrix includes Windows, use one of these patterns: (a) split into separate `run:` steps (one command per step), or (b) explicit `&&` chaining on one line (works under both shells), or (c) `shell: bash` on the step (forces bash on Windows runners too — Git Bash is available). Of these, (a) gives the cleanest failure-mode surface — every step's pass/fail is visible in the UI individually.

### 5.3 GitHub Actions empty-secret rendering breaks electron-builder Mac signing path

When a repo secret is missing, `${{ secrets.X }}` evaluates to an empty string at runtime. The env var IS set in the runner shell — to `''`. JavaScript truthy checks on `process.env.X` return `false`, but **electron-builder's signing logic does not always use a strict truthy check**. Empty `CSC_LINK` is treated as "use default cert path" → falls back to cwd → fails when cwd is a directory.

**Fix pattern:** for any electron-builder Mac config that depends on signing secrets, set `mac.identity: null` explicitly in `electron-builder.yml` as a deterministic disabler. When real secrets land, remove the line and the env-var scaffolding takes over. Don't rely on electron-builder's empty-string fallback behavior — it's not contractually documented and apparently changes across versions.

### 5.4 `actions/download-artifact merge-multiple: true` silently overwrites same-named files

This is the SHA256SUMS bug's root cause. `merge-multiple: true` is documented as "merge artifacts into a single directory". What "merge" means in conflict cases is **last-write-wins**, with no warning. If two artifact bundles each contain a `foo.txt`, only one survives extraction.

**Heuristic:** any artifact filename that appears identically across multiple matrix jobs gets clobbered on merge. Either (a) prefix the filename per-matrix-leg in the producing step (`SHA256SUMS-${{ matrix.platform }}.txt`), or (b) generate the cross-job file in the consuming job AFTER merge (the v3.2.1 follow-up plan).

### 5.5 The `[Unreleased]` section in CHANGELOG drifts; cross-check before tagging

The v3.2.0 entry's `### Removed` section accurately reflected the state of `integration-guide.md` when the [Unreleased] section was first written (deleted in `589913a`). It went stale when Rocky restored the file in `857d3c0`, but I had already moved the [Unreleased] content into [3.2.0] before the restore. The result was a CHANGELOG entry that contradicted the actual repo state. `b641e7d` corrected it.

**Heuristic:** between writing the version-N entry and pushing the version-N tag, re-grep CHANGELOG.md for every file path it mentions and verify the file state matches the claim (`git ls-files <path>` for added/changed, absence for deleted). If you run this check right before `git tag`, you catch drift like this without needing a separate validation pass.

### 5.6 Parallel work on a shared working tree is fine — but commit attribution + work-in-flight handoffs need explicit awareness

Rocky and Claude share the same local working tree (same `Strategia-Enhanced-App/Team-X/` path, no per-agent worktree). Rocky's `git commit --amend` of `a6297ef` → `857d3c0` happened between two of my Claude operations and silently dropped my staged release.yml + electron-builder.yml edits back into the working tree. I detected the change via `git log` and recovered the edits in `b641e7d`.

**Heuristic:** check `git log -1` and `git diff` before assuming the working tree state matches what you last left it in. If something feels off (a staged change disappears, HEAD moves without your push), don't proceed — investigate first. The fix is usually clean (re-stage, re-commit with proper attribution), but only if you notice.

### 5.7 Manual draft cleanup is a valid release-pipeline option

Three of the four bugs encountered this session were fixable on the live draft release using `gh release` commands — without re-running the workflow. The per-OS artifact bundles persist for 7 days, so SHA256SUMS reconstruction was straightforward. Deleting a stray asset is one `gh release delete-asset` call. Publishing is `gh release edit v3.2.0 --draft=false --latest`.

**Heuristic:** when release.yml produces a draft with fixable defects and the underlying installers are correct, manual draft cleanup is faster and less risky than another full workflow run. Reserve workflow re-runs for cases where the installers themselves are wrong.
